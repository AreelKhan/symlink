# Decision Log

| # | Decision | Why | Date |
|---|----------|-----|------|
| 1 | Web client is online-only (no offline/PWA support) | Reduces scope significantly — no service worker, no OPFS, no wa-sqlite needed for web. Desktop/mobile (Tauri) handle offline natively. Can add PWA capabilities later without major refactoring since projection logic lives in `packages/shared` as TypeScript. | 2026-03-29 |
| 2 | RESOLVED: Use Drizzle with `sqlite-proxy` + Tauri SQL plugin for client DB access | Resolves the tension from the earlier open decision. Drizzle's `sqlite-proxy` mode lets you provide a custom callback that receives generated SQL + params — we route that through `@tauri-apps/plugin-sql` which handles IPC to Rust. Rust executes natively with rusqlite and returns rows. TypeScript gets full type safety from Drizzle's query builder, native SQLite performance from Rust, and one schema definition for models/migrations. Pattern validated by Tauri community contributors (ReiLoko) and acknowledged by Tauri working group (FabianLars). Known minor issues: `.returning()` doesn't work yet, and explicit snake_case column names needed. Migrations run through the same proxy. | 2026-04-02 |
| 3 | Defer web app entirely — build desktop/mobile (Tauri) only for now | Building a web client introduces a second code path (online-first vs offline-first) that adds complexity without immediate value. Focus on the Tauri app which covers desktop and mobile. Web can be added later — the architecture supports it since all schemas and business logic live in `packages/shared` as TypeScript. | 2026-04-02 |
| 4 | SQLite Drizzle schema is the canonical model definition — no separate TypeScript interfaces for data models | The SQLite schema (used by the primary Tauri client) defines both the table and the type via `$inferSelect`. PostgreSQL schema must match, enforced by a compile-time type assertion (`satisfies`) and CI check. This means model changes require exactly two updates: (1) SQLite schema, (2) PostgreSQL schema. No standalone TypeScript interfaces for data models — they'd be a third thing to maintain with no consumer. | 2026-04-02 |
| 5 | Defer `userId` on the event model — each user gets a separate SQLite database file | The database file itself is the user partition (`symlink_{userId}.db`). No need for `userId` column or per-query filtering. On login/switch, re-point the DB connection. Add `userId` to events later if server-side needs demand it. | 2026-04-05 |
| 6 | Defer `deviceId` on the event model | Device ID is embedded in the HLC node field. If we later need it for direct queries, we can denormalize it into its own column. | 2026-04-05 |
| 7 | Event `type` is text, not enum — use TypeScript discriminated unions for type safety | SQLite has no native enum. Adding a CHECK constraint would require a migration every time a new event type is added. Type safety comes from discriminated unions on the TS side, not the DB column. | 2026-04-05 |
| 8 | `entityId` nullability enforced in TypeScript, not the DB | The DB column is nullable text. TypeScript discriminated unions enforce that entity-bound events require a non-null `entityId` and system events allow null. | 2026-04-05 |
| 9 | Defer event `version` field | Not needed until we have deployed production event schemas that require migration. Easy to add later. | 2026-04-05 |
| 10 | Only the event store module may read/write the events table | Business logic uses `appendEvent()` and (later) `markReverted()`. No raw `db.update(events)` or `db.insert(events)` outside the event store. This enforces append-only immutability at the application layer. | 2026-04-05 |
